# Story 3.1: Backend Client Management API

## Status
Draft

## Story
**As a** developer,
**I want** complete client management API endpoints with validation and audit trails,
**so that** I can support client CRUD operations with proper data integrity.

## Acceptance Criteria
1. Create API endpoints: GET, POST, PUT, DELETE `/api/clients`
2. Implement Client and Contact models with relationships
3. Add validation for required fields (name, status, relation type) and unique constraints
4. Create ClientHistory table for audit logging
5. Prevent deletion if active projects exist
6. Implement client search and filtering capabilities
7. Add proper error handling and status codes

## Dev Notes

### Database Schema Requirements
[Source: architecture/database-schema.md#client-management-tables]
- **clients table**: id, name (unique), status (active/inactive), relation_type (Customer/Partner/Internal), project_management_tool, comments, timestamps, created_by
- **client_contacts table**: id, client_id (foreign), name, email, phone, role, is_primary, created_at
- **client_history table**: id, client_id, action (created/updated/deleted/activated/deactivated), changed_fields (JSON), old_values (JSON), new_values (JSON), changed_by, changed_at
- Implement proper indexes on status, relation_type, and foreign keys

### Client Validation Rules
[Source: docs/prd/epic-3-client-management.md]
- **Required fields**: Client Name, Status, Relation Type
- **Optional fields**: Project management tool selection, comments
- **Unique constraints**: Client name must be unique across system
- **Status values**: 'active', 'inactive'
- **Relation types**: 'Customer', 'Partner', 'Internal'
- **Deactivation rules**: Requires confirmation and project dependency check

### API Endpoint Specifications
[Source: architecture/coding-standards.md#backend-standards]
- **GET /api/clients**: List clients with pagination, search, and filtering
- **POST /api/clients**: Create new client with validation
- **GET /api/clients/{id}**: Retrieve single client with contacts
- **PUT /api/clients/{id}**: Update client information
- **DELETE /api/clients/{id}**: Delete client (with dependency check)
- **GET /api/clients/{id}/history**: Retrieve client audit history
- **GET/POST/PUT/DELETE /api/clients/{id}/contacts**: Contact management

### Role-Based Access Control
[Source: docs/prd/epic-2-authentication-authorization.md]
- **HR users**: Full access to all client operations
- **PC users**: Read/write access to client data and management
- **RM users**: Read-only access to client information
- Implement endpoint-level permission checking using auth middleware

### Data Models Structure
[Source: architecture/source-tree.md#backend-structure]
- **Models**: `backend/app/models/client.py` - Client, ClientContact, ClientHistory SQLAlchemy models
- **Schemas**: `backend/app/schemas/client.py` - Pydantic models for client requests/responses
- **Services**: `backend/app/services/client_service.py` - Client business logic and validation
- **API**: `backend/app/api/clients.py` - Client management endpoints

### Search and Filtering Implementation
Query parameters for GET /api/clients:
```
?search=<text>          # Search in name and comments
&status=<active|inactive>  # Filter by status
&relation_type=<Customer|Partner|Internal>  # Filter by type
&page=<number>          # Pagination page number
&page_size=<number>     # Items per page (default 20, max 100)
&sort_by=<field>        # Sort field (name, created_at, updated_at)
&sort_order=<asc|desc>  # Sort direction
```

### Audit Trail Format
ClientHistory JSON structure:
```json
{
  "action": "updated",
  "changed_fields": ["status", "comments"],
  "old_values": {
    "status": "active",
    "comments": "Old comment"
  },
  "new_values": {
    "status": "inactive", 
    "comments": "Updated comment"
  },
  "changed_by": 1,
  "changed_at": "2025-08-20T10:30:00Z"
}
```

### Business Logic Rules
- Client deletion only allowed if no active projects exist
- Client deactivation requires confirmation dialog
- Contact email addresses should be validated but not required to be unique
- Primary contact designation: only one per client
- Client name uniqueness enforced at database level

### Error Response Format
[Source: architecture/coding-standards.md#error-handling]
```json
{
  "status": "error",
  "message": "Validation failed",
  "code": "CLIENT_001", 
  "details": {
    "field_errors": {
      "name": ["Client name already exists"],
      "email": ["Invalid email format"]
    }
  }
}
```
