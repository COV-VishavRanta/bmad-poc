### FTE Allocation Rules & Constraints
Based on PRD requirements:

| Constraint Type | Rule | Validation | Example |
|-----------------|------|------------|---------|
| **Individual Capacity** | Max 1.0 FTE per person | Sum of all active assignments ≤ 1.0 | Person A: 0.6 + 0.4 = 1.0 ✓ |
| **Date Overlap** | No over-allocation during overlaps | Date intersection FTE validation | Feb-Jun overlap: 0.5 + 0.5 = 1.0 ✓ |
| **Project Uniqueness** | One active assignment per person per project | Unique constraint per person-project | Person A can't have 2 roles on Project X |
| **SOW Alignment** | Assignments match SOW roles | Role type validation against SOW | Developer assignment needs Developer SOW role |

### Database Schema Design
```sql
-- Team Assignments table
team_assignments (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  project_id BIGINT NOT NULL,
  role_type ENUM('Developer', 'Tester', 'Business_Analyst', 'Architect', 'Designer') NOT NULL,
  role_level ENUM('Junior', 'Mid', 'Senior', 'Lead') NOT NULL,
  fte DECIMAL(3,2) NOT NULL CHECK (fte BETWEEN 0.1 AND 1.0),
  start_date DATE NOT NULL,
  end_date DATE NOT NULL,
  status ENUM('Active', 'Planned', 'Completed', 'Cancelled') NOT NULL DEFAULT 'Planned',
  hourly_rate DECIMAL(8,2),
  notes TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  created_by BIGINT NOT NULL,
  updated_by BIGINT,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (created_by) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id),
  UNIQUE KEY unique_user_project_active (user_id, project_id, status) WHERE status = 'Active',
  CHECK (start_date <= end_date)
)

-- Assignment History table
assignment_history (
  id BIGINT PRIMARY KEY,
  assignment_id BIGINT NOT NULL,
  action ENUM('CREATE', 'UPDATE', 'DELETE', 'STATUS_CHANGE', 'FTE_CHANGE') NOT NULL,
  old_values JSON,
  new_values JSON,
  fte_impact DECIMAL(3,2),
  overlap_analysis JSON,
  changed_by BIGINT NOT NULL,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (assignment_id) REFERENCES team_assignments(id),
  FOREIGN KEY (changed_by) REFERENCES users(id)
)

-- FTE Capacity View (for quick capacity checking)
CREATE VIEW user_fte_capacity AS
SELECT 
  user_id,
  SUM(fte) as total_fte,
  COUNT(*) as active_assignments,
  MIN(start_date) as earliest_start,
  MAX(end_date) as latest_end
FROM team_assignments 
WHERE status = 'Active' 
GROUP BY user_id;
```

### FTE Overlap Detection Algorithm
```python
def validate_fte_overlap(user_id, start_date, end_date, fte, exclude_assignment_id=None):
    """
    Algorithm for detecting FTE overlaps:
    1. Get all active assignments for user in date range
    2. For each day in assignment period:
       - Calculate total FTE for that day
       - Ensure total ≤ 1.0
    3. Return validation result with conflict details
    """
    
    overlapping_assignments = get_overlapping_assignments(
        user_id, start_date, end_date, exclude_assignment_id
    )
    
    for assignment in overlapping_assignments:
        overlap_start = max(start_date, assignment.start_date)
        overlap_end = min(end_date, assignment.end_date)
        
        if overlap_start <= overlap_end:
            total_fte = fte + assignment.fte
            if total_fte > 1.0:
                return ValidationError(
                    f"FTE overlap of {total_fte} during {overlap_start} to {overlap_end}"
                )
    
    return ValidationSuccess()
```

### Validation Rules
- **Required Fields**: User ID, Project ID, Role Type, Role Level, FTE, Start Date, End Date
- **FTE Range**: 0.1 to 1.0 with 2 decimal precision
- **Business Rules**:
  - Maximum 1.0 FTE per person across all active assignments
  - No date overlaps that exceed 1.0 FTE total
  - Only one active assignment per person per project
  - Assignments must align with SOW role definitions
  - Start date must be before or equal to end date
  - Cannot assign to inactive projects or users

### Role-Based Access Control
- **HR Role**: Full CRUD access to all assignments
- **RM Role**: Full CRUD access to assignments, primary assignment manager
- **PC Role**: View and edit assignments for managed projects

### Assignment Status Lifecycle
- **Planned**: Future assignment not yet started
- **Active**: Currently active assignment
- **Completed**: Assignment finished successfully
- **Cancelled**: Assignment terminated before completion

### SOW Integration Rules
- **Role Matching**: Assignment role must exist in project's SOW
- **Capacity Checking**: Total assignments should not exceed SOW allocations
- **Availability Validation**: SOW must have available FTE for the role
- **Level Compatibility**: Assignment level should match SOW requirements
