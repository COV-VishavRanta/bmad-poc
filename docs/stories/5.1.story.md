### Database Schema Requirements
[Source: architecture/database-schema.md#project-management-tables]
Group model structure and relationships:
- **groups table**: id, name, client_id (foreign), sow_id (foreign), start_date, end_date, description, status, created_at, updated_at, created_by

### Group Validation Rules
[Source: docs/prd/epic-5-group-management.md]
Business validation requirements:
- **Required fields**: Client ID, Group Name, Start Date, End Date
- **Unique naming**: Group names must be unique within each client
- **Date constraints**: Group start_date <= end_date, group dates must contain all project dates
- **SOW validation**: SOW must belong to the same client as the group (optional field)
- **Project containment**: All projects in group must have dates within group date range

### Date Containment Validation Logic
Group date containment enforcement:
```python
def validate_group_project_dates(group: Group, project: Project) -> bool:
    """Validate that project dates fall within group date boundaries"""
    if group.start_date > project.start_date:
        raise ValidationError("Project start date must be after group start date")
    if group.end_date < project.end_date:
        raise ValidationError("Project end date must be before group end date")
    return True

def validate_group_date_update(group_id: int, new_start: date, new_end: date) -> bool:
    """Validate group date changes don't violate existing project constraints"""
    projects = get_group_projects(group_id)
    for project in projects:
        if new_start > project.start_date or new_end < project.end_date:
            raise ValidationError(f"Group date change would violate project {project.name} dates")
    return True
```

### Business Logic Rules and Constraints
Key business rules for group management:
- Group deletion only allowed if no active projects are assigned
- Group name changes must maintain uniqueness within client scope
- Group date modifications must not violate existing project date ranges
- SOW changes require validation that all projects remain valid
- Group status changes must consider status of contained projects

### Integration Points
Group module integration with other systems:
- **Client Module**: Groups belong to clients, client deactivation affects groups
- **SOW Module**: Optional group-SOW association for resource planning
- **Project Module**: Projects can be optionally grouped with date constraints
- **Assignment Module**: Group-level assignment planning and resource allocation
- **User Module**: Group creation and updates tracked by user
