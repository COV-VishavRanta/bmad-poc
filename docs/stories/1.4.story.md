# Story 1.4: Project Management Module - Core CRUD Operations

## Status
Draft

## Story
**As a** PC/HR user,
**I want** a complete project management system with CRUD operations, client associations, and audit history,
**so that** I can manage project lifecycles effectively with proper client relationships and comprehensive tracking for business operations.

## Acceptance Criteria

1. **Documentation First**: Complete project management documentation created BEFORE any code implementation
2. **Project CRUD**: Full Create, Read, Update, Delete operations for projects
3. **Client Association**: Projects must be linked to existing clients with validation
4. **Date Validation**: Start date must be before end date with proper validation
5. **Audit Trail**: Complete audit history for all project changes
6. **Project Types**: Support for different project categories and types
7. **Role-Based Access**: PC can manage projects, HR has full access, RM has read access
8. **Project API Documentation**: Complete API documentation with examples and error handling

### Database Schema Design
```sql
-- Projects table
projects (
  id BIGINT PRIMARY KEY,
  client_id BIGINT NOT NULL,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  project_type ENUM('Development', 'Consulting', 'Maintenance') NOT NULL,
  project_category ENUM('Individual', 'Group') NOT NULL DEFAULT 'Individual',
  status ENUM('Planning', 'Active', 'On-Hold', 'Completed', 'Cancelled') NOT NULL DEFAULT 'Planning',
  start_date DATE NOT NULL,
  end_date DATE NOT NULL,
  budget DECIMAL(15,2),
  group_id BIGINT NULL,
  sow_id BIGINT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  created_by BIGINT NOT NULL,
  updated_by BIGINT,
  FOREIGN KEY (client_id) REFERENCES clients(id),
  FOREIGN KEY (group_id) REFERENCES project_groups(id),
  FOREIGN KEY (sow_id) REFERENCES sows(id),
  FOREIGN KEY (created_by) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id),
  UNIQUE KEY unique_project_name_per_client (client_id, name),
  CHECK (start_date <= end_date)
)

-- Project audit history
project_history (
  id BIGINT PRIMARY KEY,
  project_id BIGINT NOT NULL,
  action ENUM('CREATE', 'UPDATE', 'DELETE', 'STATUS_CHANGE') NOT NULL,
  old_values JSON,
  new_values JSON,
  changed_by BIGINT NOT NULL,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES projects(id),
  FOREIGN KEY (changed_by) REFERENCES users(id)
)
```

### API Endpoints Specification
```
GET /api/projects
  - Query params: page, limit, client_id, status, project_type, search
  - Response: paginated project list with client info

GET /api/projects/{id}
  - Response: single project with client details and audit history

POST /api/projects
  - Body: project data (client_id, name, type, dates, etc.)
  - Response: created project with ID

PUT /api/projects/{id}
  - Body: updated project data
  - Response: updated project with audit entry

DELETE /api/projects/{id}
  - Checks: no active team assignments exist
  - Response: deletion confirmation or dependency error
```

### Validation Rules
- **Required Fields**: Client ID, Project Name, Project Type, Start Date, End Date
- **Optional Fields**: Description, Budget, Group ID, SOW ID
- **Business Rules**:
  - Project names must be unique within each client
  - Start date must be before or equal to end date
  - Client must exist and be active
  - Budget must be positive if provided
  - Cannot delete projects with active team assignments

### Role-Based Access Control
- **HR Role**: Full CRUD access to all projects
- **PC Role**: Full CRUD access to projects, can assign to any client
- **RM Role**: Read-only access to project data for resource planning

### Audit Trail Requirements
- Track all create, update, delete, and status change operations
- Store old and new values for all changes
- Include user who made the change and timestamp
- Provide audit history in project detail views

### Project Status Lifecycle
- **Planning**: Initial project setup phase
- **Active**: Project is currently running
- **On-Hold**: Temporarily paused
- **Completed**: Successfully finished
- **Cancelled**: Terminated before completion

### MVP Requirements

- **Frontend**: Basic project listing and forms with client selection
- **Database**: Proper schema with client relationships and constraints
