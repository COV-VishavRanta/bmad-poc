## Status

## Story
**As a** developer,
**I want** a properly configured database with ORM and migrations,
**so that** I can manage data models and schema changes effectively.

## Dev Notes

### Database Schema Requirements
[Source: architecture/database-schema.md]
The complete database schema includes:
- **User management**: users, user_sessions tables with role-based access
- **Client management**: clients, client_contacts, client_history tables
- **Project management**: projects, groups, project_history tables  
- **SOW management**: sows, sow_roles, sow_history tables
- **Assignment management**: assignments, assignment_history tables
- All tables include proper indexes, foreign keys, and constraint validation

### SQLAlchemy Configuration
[Source: architecture/tech-stack.md#database-and-orm]
- **SQLAlchemy 2.0+**: Use modern SQLAlchemy with new query syntax
- **MySQL Driver**: pymysql for Python MySQL connectivity
- **Connection Pooling**: Configure pool_size=20, max_overflow=0, pool_recycle=3600
- **Session Management**: Use session-per-request pattern with FastAPI
- **Async Support**: Configure for potential async operations

### Database Connection Configuration
Environment variables from .env:
```
# Database Configuration
MYSQL_DATABASE=clientops_db
MYSQL_USER=clientops_user
MYSQL_PASSWORD=secure_password
MYSQL_ROOT_PASSWORD=root_password
DATABASE_URL=mysql+pymysql://clientops_user:secure_password@mysql:3306/clientops_db

# Connection Pool Settings
DB_POOL_SIZE=20
DB_MAX_OVERFLOW=0
DB_POOL_RECYCLE=3600
DB_POOL_TIMEOUT=30
```

### Database Health Check Implementation
Create endpoint for monitoring database connectivity:
```python
@router.get("/health/database")
async def database_health_check(db: Session = Depends(get_db)):
    try:
        # Simple query to test connection
        db.execute(text("SELECT 1"))
        return {"status": "healthy", "database": "connected"}
    except Exception as e:
        raise HTTPException(
            status_code=503, 
            detail=f"Database connection failed: {str(e)}"
        )
```

### Migration Strategy
- **Initial Migration**: Create all tables in the first migration
- **Incremental Changes**: Separate migrations for each schema change
- **Data Migrations**: Include data transformation scripts when needed
- **Rollback Safety**: Ensure all migrations can be safely rolled back
- **Testing**: Test migrations on copy of production data

### Model Relationships
Key relationships to implement:
- User → UserSession (one-to-many)
- Client → ClientContact (one-to-many)
- Client → Project (one-to-many)
- Project → Assignment (one-to-many)
- User → Assignment (one-to-many)
- SOW → Project (one-to-many)
- Group → Project (one-to-many)

### Database Constraints and Validation
[Source: architecture/database-schema.md#database-constraints-and-rules]
- **FTE Constraints**: Assignment FTE between 0.1 and 1.0
- **Date Constraints**: start_date <= end_date for projects and assignments
- **Unique Constraints**: Client names, user emails, group names per client
- **Foreign Key Constraints**: Proper referential integrity
- **Check Constraints**: Status enums, role validations

### Performance Considerations
- **Indexes**: Add indexes on frequently queried fields (email, status, dates)
- **Connection Pooling**: Optimize for expected concurrent users
- **Query Optimization**: Use eager loading for common relationship queries
- **Batch Operations**: Implement efficient bulk operations for data import

- Unit tests for each model and relationship

- Migration testing with sample data
- Connection pool and session management tests
- Performance tests with realistic data volumes
