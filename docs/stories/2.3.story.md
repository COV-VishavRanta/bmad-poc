## Dev Notes

### Role-Based Access Control
[Source: docs/prd/epic-2-authentication-authorization.md]
- **HR users**: Full access to user management functionality
- **PC/RM users**: No access to user management (read-only access to user lists for assignments)
- Implement strict permission checking at both frontend and backend levels
- All user management operations must be logged in audit trail

### User Management API Specifications
Backend endpoints for user management:
- **GET /api/users**: List all users with search, filtering, and pagination
- **POST /api/users**: Create new user account
- **GET /api/users/{id}**: Get detailed user information
- **PUT /api/users/{id}**: Update user information
- **DELETE /api/users/{id}**: Deactivate user (soft delete)
- **PUT /api/users/{id}/role**: Change user role
- **PUT /api/users/{id}/status**: Change user status
- **GET /api/users/{id}/history**: Get user audit history
- **POST /api/users/bulk**: Bulk user operations

### User Model and Validation
[Source: architecture/database-schema.md#user-management-tables]
User model fields and validation:
- **Required**: email (unique), full_name, role (HR/PC/RM)
- **Optional**: phone, department, hire_date, profile_image
- **System**: password_hash, status, created_at, updated_at, last_login
- **Validation**: Email format, role enum, password complexity requirements

### User Interface Design
[Source: docs/Web-app-prompt 1#user-interface-design]
User management interface components:
- User data table with sortable columns and search
- User creation/editing modal dialogs
- Role assignment dropdown with permission previews
- User status toggle switches with confirmations
- Bulk operation selection and action buttons
- User audit history timeline display

### Search and Filtering Implementation
User search capabilities:
```typescript
interface UserSearchParams {
  search?: string          // Search in name, email
  role?: 'HR' | 'PC' | 'RM' | 'all'
  status?: 'active' | 'inactive' | 'all'
  page?: number
  pageSize?: number
  sortBy?: 'name' | 'email' | 'role' | 'created_at' | 'last_login'
  sortOrder?: 'asc' | 'desc'
  dateFrom?: string        // Filter by creation date
  dateTo?: string
}
```

### User Creation and Invitation Flow
User creation process:
1. HR user fills out user creation form
2. System generates temporary password
3. User invitation email sent with login credentials
4. New user must change password on first login
5. User account activated upon first successful login
6. All actions logged in audit trail

### Password Management
[Source: docs/stories/2.1.story.md#password-security]

- Implement password complexity requirements
- Force password change on first login
- Provide password reset functionality for existing users
- Log all password-related activities
