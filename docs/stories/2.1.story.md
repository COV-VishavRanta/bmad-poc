# Story 2.1: Backend Authentication Infrastructure

## Status
Draft

## Story
**As a** developer,
**I want** secure session management and role-based middleware,
**so that** I can protect API endpoints and manage user authentication securely.

## Acceptance Criteria
1. Implement secure session management with HttpOnly cookies
2. Create FastAPI middleware for role-based endpoint protection
3. Add token expiration and refresh logic
4. Define RBAC policies for each endpoint
5. Create demo users for each role (HR, PC, RM)
6. Implement password hashing and validation
7. Create authentication service with login/logout functionality

## Dev Notes

### Database Schema Requirements
[Source: architecture/database-schema.md#user-management-tables]
- **users table**: id, email (unique), password_hash, full_name, role (HR/PC/RM), status (active/inactive), timestamps
- **user_sessions table**: session_id (primary), user_id (foreign), expires_at, ip_address, user_agent, timestamps
- Use proper indexes on email, role, status, and session expiration fields
- Implement foreign key constraints and proper data types

### Password Security
[Source: architecture/coding-standards.md#security-standards]
- Use passlib with bcrypt algorithm for password hashing
- Minimum password requirements: 8 characters, mixed case, numbers
- Implement secure password verification with timing attack protection
- Store only hashed passwords, never plain text

### Session Management
[Source: architecture/tech-stack.md#authentication-and-security]
- Use HttpOnly cookies to prevent XSS attacks
- Set Secure flag for HTTPS environments
- Implement SameSite=Lax for CSRF protection
- Default session expiration: 24 hours (1440 minutes)
- Store session data server-side, only session ID in cookie

### Role Definitions and Permissions
[Source: docs/prd/epic-2-authentication-authorization.md]
- **HR (Human Resources)**: Full system access - manage users, view all data
- **PC (Project Coordinator)**: Manage projects, view clients, access SOWs
- **RM (Resource Manager)**: Manage assignments, view project timelines and team data
- Implement permission matrix for endpoint access control

### FastAPI Implementation Patterns
[Source: architecture/coding-standards.md#backend-standards]
- Use dependency injection for database sessions and user context
- Implement Pydantic models for request/response validation
- Create custom middleware classes for authentication processing
- Use FastAPI's built-in exception handling for auth errors
- Follow REST API conventions for endpoint design

### File Structure
[Source: architecture/source-tree.md#backend-structure]
- **Models**: `backend/app/models/user.py` - User and UserSession SQLAlchemy models
- **Schemas**: `backend/app/schemas/auth.py` - Pydantic models for auth requests/responses
- **Services**: `backend/app/services/auth_service.py` - Authentication business logic
- **API**: `backend/app/api/auth.py` - Authentication endpoints
- **Utils**: `backend/app/utils/security.py` - Password hashing and session utilities
- **Middleware**: `backend/app/middleware.py` - RBAC and auth middleware

### API Response Formats
[Source: architecture/coding-standards.md#error-handling]
- Success responses: `{"status": "success", "data": {...}}`
- Error responses: `{"status": "error", "message": "...", "code": "AUTH_001"}`
- Consistent HTTP status codes: 200 (success), 401 (unauthorized), 403 (forbidden), 422 (validation error)
