# Story 1.6: Statement of Work (SOW) Module - Resource Definition & FTE Management

## Status
Draft

## Story
**As a** PC/HR user,
**I want** a comprehensive SOW management system with role-based FTE allocations and versioned history,
**so that** I can define project resource requirements accurately and track resource planning changes over time for effective project staffing.

1. **Documentation First**: Complete SOW management documentation created BEFORE any code implementation
2. **SOW CRUD**: Full Create, Read, Update, Delete operations for SOWs
3. **Role Templates**: Support for predefined roles (Developer, Tester, Business Analyst, Architect, Designer)
4. **FTE Allocation**: Ability to define FTE requirements per role with validation
5. **SOW Versioning**: Complete version history for SOW changes and updates
6. **Project Constraint**: One active SOW per project with proper validation
7. **Role-Based Access**: PC can manage SOWs, HR has full access, RM can view for planning
8. **SOW API Documentation**: Complete API documentation with examples and error handling

### Database Schema Design
```sql
-- SOWs table
sows (
  id BIGINT PRIMARY KEY,
  client_id BIGINT NOT NULL,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  version VARCHAR(50) NOT NULL DEFAULT '1.0',
  status ENUM('Draft', 'Active', 'Inactive', 'Archived') NOT NULL DEFAULT 'Draft',
  start_date DATE,
  end_date DATE,
  total_fte DECIMAL(5,2) GENERATED ALWAYS AS (
    SELECT SUM(fte) FROM sow_role_allocations WHERE sow_id = sows.id
  ) STORED,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  created_by BIGINT NOT NULL,
  updated_by BIGINT,
  FOREIGN KEY (client_id) REFERENCES clients(id),
  FOREIGN KEY (created_by) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id),
  UNIQUE KEY unique_active_sow_per_project (client_id, status) WHERE status = 'Active'
)

-- SOW Role Allocations table
sow_role_allocations (
  id BIGINT PRIMARY KEY,
  sow_id BIGINT NOT NULL,
  role_type ENUM('Developer', 'Tester', 'Business_Analyst', 'Architect', 'Designer') NOT NULL,
  role_level ENUM('Junior', 'Mid', 'Senior', 'Lead') NOT NULL DEFAULT 'Mid',
  fte DECIMAL(4,2) NOT NULL CHECK (fte > 0 AND fte <= 10.0),
  hourly_rate DECIMAL(8,2),
  description TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sow_id) REFERENCES sows(id) ON DELETE CASCADE,
  UNIQUE KEY unique_role_per_sow (sow_id, role_type, role_level)
)

-- SOW History table
sow_history (
  id BIGINT PRIMARY KEY,
  sow_id BIGINT NOT NULL,
  version VARCHAR(50) NOT NULL,
  action ENUM('CREATE', 'UPDATE', 'ACTIVATE', 'DEACTIVATE', 'ARCHIVE') NOT NULL,
  old_values JSON,
  new_values JSON,
  role_allocations JSON,
  changed_by BIGINT NOT NULL,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  start_date DATE,
  end_date DATE,
  FOREIGN KEY (sow_id) REFERENCES sows(id),
  FOREIGN KEY (changed_by) REFERENCES users(id)
)

-- Update projects table to include SOW reference
ALTER TABLE projects ADD COLUMN sow_id BIGINT NULL;
ALTER TABLE projects ADD FOREIGN KEY (sow_id) REFERENCES sows(id);
```

### API Endpoints Specification
```
GET /api/sows
  - Query params: page, limit, client_id, status, project_id, search
  - Response: paginated SOW list with total FTE and role counts

GET /api/sows/{id}
  - Response: single SOW with role allocations and history

GET /api/sows/{id}/history
  - Response: SOW version history with changes

POST /api/sows
  - Body: SOW data with role allocations array
  - Response: created SOW with role allocations

PUT /api/sows/{id}
  - Body: updated SOW data (creates new version)
  - Response: updated SOW with new version number

DELETE /api/sows/{id}
  - Checks: no active project assignments
  - Response: deletion confirmation or dependency error

POST /api/sows/{id}/roles
  - Body: role allocation data
  - Response: added role allocation

PUT /api/sows/{id}/roles/{role_id}
  - Body: updated role allocation
  - Response: updated role allocation

DELETE /api/sows/{id}/roles/{role_id}
  - Response: role removal confirmation
```

### FTE Calculation & Validation Logic
```
FTE Validation Rules:
1. Individual role FTE: 0.1 - 10.0 (allows for team leads managing multiple people)
2. Total SOW FTE: sum of all role allocations
3. FTE must be positive decimal with 2 decimal places
4. Role allocations must have valid role type and level
5. Duplicate role+level combinations not allowed within same SOW
```

### Validation Rules
- **Required Fields**: Client ID, SOW Name, at least one role allocation
- **Optional Fields**: Description, Start Date, End Date, Hourly Rates
- **Business Rules**:
  - Only one active SOW per project at any time
  - Cannot delete SOW if assigned to active projects
  - Version numbers increment automatically on updates
  - Role allocations must have positive FTE values
  - Cannot activate SOW without role allocations

### Role-Based Access Control
- **HR Role**: Full CRUD access to all SOWs and role allocations
- **PC Role**: Full CRUD access to SOWs, can manage role allocations
- **RM Role**: Read access to all SOWs for resource planning purposes

### SOW Versioning Logic
- **Automatic Versioning**: Every SOW update creates new version
- **Version Format**: Major.Minor (1.0, 1.1, 2.0, etc.)
- **Version Triggers**: Role allocation changes, date changes, status changes
- **History Retention**: Complete history with old/new values for all changes

### SOW Status Lifecycle
- **Draft**: Initial creation, can be edited freely
- **Active**: Currently being used for project planning
- **Inactive**: Temporarily disabled
- **Archived**: Completed or obsolete SOWs
