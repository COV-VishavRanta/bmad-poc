### Form Validation Schema
Project form validation using Zod:
```typescript
const projectFormSchema = z.object({
  name: z.string().min(1, 'Project name is required').max(255),
  clientId: z.number().min(1, 'Client selection is required'),
  sowId: z.number().min(1, 'SOW selection is required'),
  projectType: z.enum(['Development', 'Maintenance', 'Consulting', 'Support']),
  status: z.enum(['planned', 'active', 'on_hold', 'completed', 'cancelled']),
  startDate: z.date(),
  endDate: z.date(),
  groupId: z.number().optional(),
  description: z.string().max(1000).optional(),
}).refine((data) => data.startDate <= data.endDate, {
  message: "End date must be after start date",
  path: ["endDate"],
})
```

### Role-Based Access Control
[Source: docs/prd/epic-2-authentication-authorization.md]
Project management permissions:
- **HR users**: Full access to all project operations
- **PC users**: Full read/write access to project management and creation
- **RM users**: Read-only access to project information for assignment context
- Implement component-level visibility and action restrictions

### API Integration
[Source: docs/stories/4.1.story.md#api-endpoint-specifications]
Frontend API calls for project management:
- **GET /api/projects**: Fetch projects with search and pagination
- **POST /api/projects**: Create new project
- **GET /api/projects/{id}**: Fetch single project with details
- **PUT /api/projects/{id}**: Update project information
- **PUT /api/projects/{id}/status**: Change project status
- **DELETE /api/projects/{id}**: Delete/archive project

### Project Search and Filter Implementation
Search parameters interface:
```typescript
interface ProjectSearchParams {
  search?: string
  clientId?: number
  status?: ProjectStatus | 'all'
  projectType?: ProjectType | 'all'
  groupId?: number
  startDateFrom?: string
  startDateTo?: string
  endDateFrom?: string
  endDateTo?: string
  page?: number
  pageSize?: number
  sortBy?: 'name' | 'start_date' | 'end_date' | 'status'

```

### Responsive Design Requirements
- **Desktop**: Full table view with all columns and detailed information
- **Tablet**: Condensed table with essential columns, expandable details
- **Mobile**: Card-based layout with swipe actions and collapsible sections
- **Touch-friendly**: Larger buttons and touch targets for mobile interaction
- **Progressive enhancement**: Advanced features available on larger screens

### Loading States and UX
- **Table loading**: Skeleton rows during project data fetch
- **Form submission**: Button loading states with progress indicators
- **Search**: Debounced input with loading indicators
- **Status changes**: Immediate optimistic updates with rollback capability
- **Navigation**: Loading states during page transitions

### Error Handling Strategy
[Source: architecture/coding-standards.md#error-handling]
- **Validation errors**: Real-time field validation with clear messaging
- **Network errors**: Automatic retry with exponential backoff
- **Permission errors**: Clear unauthorized access messaging
- **Conflict errors**: User-friendly conflict resolution options
- **Form errors**: Comprehensive error summary and field highlighting

### Performance Optimization
- **Virtual scrolling**: Handle large project lists efficiently
- **Debounced search**: Prevent excessive API calls during search
- **Optimistic updates**: Immediate UI feedback for better UX
- **Data caching**: Cache project data with proper invalidation
- **Lazy loading**: Load project details and related data on demand

### Accessibility Features
- **Keyboard navigation**: Full keyboard support for all operations
- **Screen reader**: Proper ARIA labels and descriptions
- **High contrast**: Support for high contrast and dark mode
- **Focus management**: Clear focus indicators and logical tab order
- **Announcements**: Screen reader announcements for status changes
