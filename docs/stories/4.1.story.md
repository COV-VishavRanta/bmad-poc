## Dev Notes

### Database Schema Requirements
[Source: architecture/database-schema.md#project-management-tables]
Project model structure:
- **projects table**: id, name, client_id (foreign), group_id (foreign), sow_id (foreign), project_type, status, start_date, end_date, description, created_at, updated_at, created_by
- **project_history table**: id, project_id, action, changed_fields (JSON), old_values (JSON), new_values (JSON), changed_by, changed_at
- Required indexes on client_id, group_id, status, dates for performance

### Project Validation Rules
[Source: docs/Web-app-prompt 1#step-4-project-management-module]
Business validation requirements:
- **Required fields**: Client ID, Project Name, Start Date, End Date, SOW ID, Project Type
- **Date constraints**: Start date must be before end date
- **SOW validation**: SOW must be active and belong to the same client
- **Group constraints**: Project dates must fall within group date range (if grouped)
- **Status transitions**: Defined workflow for status changes

### Project Status Management
Project status enum and transitions:
```python
class ProjectStatus(str, Enum):
    PLANNED = "planned"
    ACTIVE = "active"
    ON_HOLD = "on_hold"
    COMPLETED = "completed"
    CANCELLED = "cancelled"

# Valid status transitions
STATUS_TRANSITIONS = {
    "planned": ["active", "cancelled"],
    "active": ["on_hold", "completed", "cancelled"],
    "on_hold": ["active", "cancelled"],
    "completed": [],  # Terminal state
    "cancelled": []   # Terminal state
}
```

### API Endpoint Specifications
[Source: architecture/coding-standards.md#backend-standards]
Project API endpoints:
- **GET /api/projects**: List projects with search, filtering, pagination
- **POST /api/projects**: Create new project with validation
- **GET /api/projects/{id}**: Retrieve project with related data
- **PUT /api/projects/{id}**: Update project information
- **DELETE /api/projects/{id}**: Soft delete project (with dependency check)
- **PUT /api/projects/{id}/status**: Change project status
- **GET /api/projects/{id}/history**: Retrieve project audit history

### Role-Based Access Control
[Source: docs/prd/epic-2-authentication-authorization.md]
Project management permissions:
- **HR users**: Full access to all project operations
- **PC users**: Full read/write access to project management
- **RM users**: Read access to projects for assignment context
- Implement endpoint-level permission checking

### Search and Filtering Implementation
Project search parameters:
```python
class ProjectSearchParams(BaseModel):
    search: Optional[str] = None          # Search in name, description
    client_id: Optional[int] = None       # Filter by client
    status: Optional[ProjectStatus] = None # Filter by status
    project_type: Optional[ProjectType] = None # Filter by type
    group_id: Optional[int] = None        # Filter by group
    start_date_from: Optional[date] = None # Date range filter
    start_date_to: Optional[date] = None
    end_date_from: Optional[date] = None
    end_date_to: Optional[date] = None
    page: int = 1
    page_size: int = 20
    sort_by: str = "name"
    sort_order: str = "asc"
```

### SOW Integration and Validation
[Source: docs/Web-app-prompt 1#step-6-statement-of-work-sow-module]
SOW linkage requirements:
- Each project must be linked to exactly one active SOW
- SOW must belong to the same client as the project
- Validate SOW status is active before project creation
- Track SOW changes that affect project resource allocation
- Implement SOW-project relationship constraints

### Project History and Audit Trail
Audit trail JSON structure:
```json
{
  "action": "status_changed",
  "changed_fields": ["status", "end_date"],
  "old_values": {
    "status": "active",
    "end_date": "2025-12-31"
  },
  "new_values": {
    "status": "completed",
    "end_date": "2025-11-15"
  },
  "changed_by": 1,
  "changed_at": "2025-08-20T10:30:00Z"
}
```

### Business Logic Rules
- Project deletion only allowed if no active assignments exist
- Status changes must follow defined transition workflows
- Group association requires date validation (project within group dates)
- SOW changes require validation of resource allocations
- Project completion automatically updates assignment statuses

### Integration Points
- **Client Module**: Project belongs to client, client deactivation affects projects
- **SOW Module**: Project linked to SOW for resource planning
- **Group Module**: Optional project grouping with date constraints
- **Assignment Module**: Projects contain team member assignments
- **User Module**: Project creation and updates tracked by user

### Error Handling and Validation
[Source: architecture/coding-standards.md#error-handling]
Project-specific error codes and handling:
```json
{
  "status": "error",
  "message": "Project validation failed",
  "code": "PROJECT_001",
  "details": {
    "field_errors": {
      "start_date": ["Start date must be before end date"],
      "sow_id": ["SOW must belong to the same client"]
    }
  }
}
```

### Performance Considerations
- Index frequently queried fields (client_id, status, dates)
- Optimize search queries with proper database indexing
- Implement efficient filtering with database-level operations
- Use eager loading for related data (client, SOW, group)
- Cache frequently accessed project data



### Data Relationships
Key project relationships to implement:
- Project → Client (many-to-one, required)
- Project → Group (many-to-one, optional)
- Project → SOW (many-to-one, required)
- Project → Assignment (one-to-many)
- Project → ProjectHistory (one-to-many)
- User → Project (many-to-many through assignments)
