# Story 1.5: Group Management Module - Project Grouping & Constraints

## Status
Draft

## Story
**As a** PC/HR user,
**I want** a project grouping system with date containment rules and unique naming constraints,
**so that** I can organize related projects into logical groups while ensuring proper date boundaries and project management oversight.

## Acceptance Criteria

1. **Documentation First**: Complete group management documentation created BEFORE any code implementation
2. **Group CRUD**: Full Create, Read, Update, Delete operations for project groups
3. **Unique Naming**: Group names must be unique within each client scope
4. **Date Containment**: Group dates must encompass all associated project dates
5. **Project Association**: Ability to assign and reassign projects to groups
6. **Audit Trail**: Complete audit history for all group changes
7. **Role-Based Access**: PC can manage groups, HR has full access, RM has read access
8. **Group API Documentation**: Complete API documentation with examples and error handling

### Group Concepts & Rules
Based on PRD requirements:

| Concept | Description | Rule |
|---------|-------------|------|
| **Group Uniqueness** | Group names within client | Unique group names per client |
| **Date Containment** | Group dates contain project dates | Group start ≤ all project starts, Group end ≥ all project ends |
| **Project Association** | Projects can belong to one group | Optional group assignment per project |

### Database Schema Design
```sql
-- Project Groups table
project_groups (
  id BIGINT PRIMARY KEY,
  client_id BIGINT NOT NULL,
  name VARCHAR(255) NOT NULL,
  description TEXT,
  start_date DATE NOT NULL,
  end_date DATE NOT NULL,
  status ENUM('Planning', 'Active', 'On-Hold', 'Completed', 'Cancelled') NOT NULL DEFAULT 'Planning',
  sow_id BIGINT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  created_by BIGINT NOT NULL,
  updated_by BIGINT,
  FOREIGN KEY (client_id) REFERENCES clients(id),
  FOREIGN KEY (sow_id) REFERENCES sows(id),
  FOREIGN KEY (created_by) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id),
  UNIQUE KEY unique_group_name_per_client (client_id, name),
  CHECK (start_date <= end_date)
)

-- Project Group audit history
project_group_history (
  id BIGINT PRIMARY KEY,
  group_id BIGINT NOT NULL,
  action ENUM('CREATE', 'UPDATE', 'DELETE', 'PROJECT_ADDED', 'PROJECT_REMOVED') NOT NULL,
  old_values JSON,
  new_values JSON,
  changed_by BIGINT NOT NULL,
  changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (group_id) REFERENCES project_groups(id),
  FOREIGN KEY (changed_by) REFERENCES users(id)
)

-- Update projects table to include group reference
ALTER TABLE projects ADD COLUMN group_id BIGINT NULL;
ALTER TABLE projects ADD FOREIGN KEY (group_id) REFERENCES project_groups(id);
```

### API Endpoints Specification
```
GET /api/groups
  - Query params: page, limit, client_id, status, search
  - Response: paginated group list with project counts

GET /api/groups/{id}
  - Response: single group with associated projects and audit history

POST /api/groups
  - Body: group data (client_id, name, dates, etc.)
  - Response: created group with ID

PUT /api/groups/{id}
  - Body: updated group data with date validation
  - Response: updated group with audit entry

DELETE /api/groups/{id}
  - Options: reassign projects or remove group association
  - Response: deletion confirmation with project impact

POST /api/groups/{id}/projects
  - Body: project IDs to assign to group
  - Response: assignment results with date validation

DELETE /api/groups/{id}/projects/{project_id}
  - Response: project removal confirmation
```

### Date Containment Logic
```
Group Date Validation Rules:
1. Group start_date ≤ MIN(project.start_date) for all projects in group
2. Group end_date ≥ MAX(project.end_date) for all projects in group
3. When adding project to group: validate dates against group boundaries
4. When updating group dates: validate against all contained project dates
5. When updating project dates: validate against group boundaries if assigned
```

### Validation Rules
- **Required Fields**: Client ID, Group Name, Start Date, End Date
- **Optional Fields**: Description, SOW ID, Status
- **Business Rules**:
  - Group names must be unique within each client
  - Group dates must contain all associated project dates
  - Cannot delete groups with active projects (unless reassigned)
  - Projects can belong to maximum one group
  - Date changes must maintain containment rules

### Role-Based Access Control
- **HR Role**: Full CRUD access to all groups
- **PC Role**: Full CRUD access to groups, can manage project assignments
- **RM Role**: Read-only access to group data for resource planning

### Audit Trail Requirements
- Track all create, update, delete operations
- Track project assignment and removal from groups
- Store old and new values for all changes
- Include user who made the change and timestamp
- Provide audit history in group detail views

### Group Status Lifecycle
- **Planning**: Initial group setup phase
- **Active**: Group is currently running with active projects
- **On-Hold**: Temporarily paused
- **Completed**: All projects in group completed
- **Cancelled**: Group terminated before completion

### MVP Requirements
- **Documentation**: Complete group management documentation before coding
- **Backend**: Full CRUD API with date containment validation and audit trails
- **Frontend**: Basic group listing and forms with project assignment
- **Database**: Proper schema with date constraints and relationships
- **Security**: Role-based access control for all operations
